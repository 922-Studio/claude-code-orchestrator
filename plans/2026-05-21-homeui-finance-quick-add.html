Current-state findings
  • Ledger empty-state tells the user to "add first entry" but no button or affordance exists anywhere on the page.
  • DebtNLInput is built and styled but not imported in App.tsx nor in any page — it currently ships as dead code.
  • Persons API + Zod schemas already exist (src/api/persons.ts, src/types/api/person.ts); no list view, no create form, no hook layer.
  • DebtCreateSchema in TS uses keys person/date but the backend POST expects person_name/transaction_date. Any structured create form must use the backend's keys; existing parser flow needs a mapping fix.
  • Backend auto-creates a Person on unknown person_name, so "add debt" implicitly covers "add person" for the common path. A standalone person-create form is only needed when Gregor wants to seed people before any debt exists (e.g. roster management).
Mockup — three entry points, one shared form

Single source of truth: a QuickAddDebtDialog component (modal). It is opened from three places:

  1. Finance overview header — primary "New debt" button next to the page title; secondary "New person" link.
  2. Ledger page header — same two buttons, plus an inline empty-state CTA when no transactions exist.
  3. Dedicated route /finance/add — same form rendered inline (not modal) for keyboard-driven / deep-link / mobile flows; useful for bookmarking and from Discord deep links.

Decisions (locked 2026-05-21)
  • "User" in scope = finance Person, not HomeAuth account. Person-create flow is the only one in this plan.
  • Currency is hard-coded EUR; remove the currency dropdown from the mockup. DebtCreate payload always sends "EUR".
  • No toast system in this PR. Success/error feedback stays inline in the form (and dialog auto-closes on success), matching the current DebtNLInput pattern. Revisit when a second feature actually needs cross-page notifications.
  • Discord bot is out of scope — no deep-link integration follow-up.
