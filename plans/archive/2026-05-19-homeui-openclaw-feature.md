## Design Decisions

### Module seed via Alembic data migration (HomeAPI)

The module catalog is persisted in Postgres. A data migration is the safest way to ship it: it runs automatically on `alembic upgrade head` during deploy and is repeatable. No manual admin API calls needed.

Migration inserts a row with `slug='openclaw'`, `name='OpenClaw'`, `description='AI agent orchestration'`, `global_status='active'`, `visible=True`, `sort_order` after the last existing module. Uses an `INSERT … ON CONFLICT DO NOTHING` guard so re-running is idempotent.

### Single OpenClaw page with four sections

Rather than sub-routes (overkill for v1), one `/openclaw` page with four distinct card sections:

| Section | Data source | Update strategy |
|---------|-------------|-----------------|
| **Status** | `GET /api/openclaw/status` | `refetchInterval: 15_000` |
| **Agent Trigger** | `POST /api/openclaw/trigger/agent` | mutation, no polling |
| **Heartbeat Monitor** | `GET /api/openclaw/state/heartbeat` | `refetchInterval: 10_000` |
| **Daily Summary** | `POST /api/openclaw/workflows/daily-summary` | manual refresh button |

The heartbeat check trigger (`POST /openclaw/workflows/heartbeat-check`) and the heartbeat state update (`PUT /openclaw/state/heartbeat`) are internal / agent-driven — not exposed in the UI in v1.

### Routing: `ModuleGuard slug="openclaw"` wrapping `OpenClawPage`

Consistent with every other module route in `App.tsx`. No sub-routes needed; the page is flat.

- **Branch**: `feat/openclaw-module-seed`
- **Worktree**: `HomeAPI/.worktrees/feat/openclaw-module-seed`
- **Parallel with**: Step 2 (independent repos)

**What to do:**

1. Run `alembic revision --autogenerate -m "seed openclaw module"` inside the worktree to create a new revision file in `alembic/versions/`.
2. The autogenerated body will be empty (no schema change). Replace the `upgrade()` body with:
   ```python
   op.execute("""
       INSERT INTO modules (id, slug, name, description, sort_order, global_status, visible, created_at, updated_at)
       VALUES (
           gen_random_uuid(),
           'openclaw',
           'OpenClaw',
           'AI agent orchestration',
           (SELECT COALESCE(MAX(sort_order), 0) + 1 FROM modules),
           'active',
           true,
           NOW(),
           NOW()
       )
       ON CONFLICT (slug) DO NOTHING
   """)
   ```
3. `downgrade()` body: `op.execute("DELETE FROM modules WHERE slug = 'openclaw'")`
4. Verify columns against the `Module` ORM model before committing — adjust field list if the model differs from what's listed above.

**Acceptance criteria:**
- [ ] Migration applies cleanly: `alembic upgrade head`
- [ ] Row exists in DB: `SELECT slug, name, global_status FROM modules WHERE slug = 'openclaw'`
- [ ] Downgrade works: `alembic downgrade -1` removes the row
- [ ] `ruff check` + `mypy` pass
- [ ] PR opened against `dev`, description references this plan

#### 2a — Types (`src/types/api/openclaw.ts`)

Create with Zod schemas matching the Pydantic shapes in `HomeAPI/app/schemas/openclaw.py`:

- `OpenClawStatusSchema` — `{ reachable: z.boolean(), status_code: z.number().nullable() }`
- `AgentTriggerRequestSchema` — `{ agent_name: z.string(), payload: z.record(z.unknown()).default({}) }`
- `AgentTriggerResponseSchema` — `{ triggered: z.boolean(), result: z.unknown() }`
- `HeartbeatSchema` — `task_name`, `first_seen_at`, `last_seen_at`, `run_count`, `status` (all strings/numbers, with `z.passthrough()` for extra fields)
- `HeartbeatMapSchema` — `z.record(HeartbeatSchema)`
- `DailySummaryResponseSchema` — `todo_count`, `overdue_count`, `summary`, `weather`

Export `z.infer<>` types for each.

#### 2b — API layer (`src/api/openclaw.ts`)

Functions:
- `getOpenClawStatus(): Promise<OpenClawStatus>` — `GET /api/openclaw/status`
- `triggerAgent(req: AgentTriggerRequest): Promise<AgentTriggerResponse>` — `POST /api/openclaw/trigger/agent`
- `getHeartbeats(): Promise<HeartbeatMap>` — `GET /api/openclaw/state/heartbeat`
- `getDailySummary(): Promise<DailySummaryResponse>` — `POST /api/openclaw/workflows/daily-summary`

`queryOptions` factories:
- `openClawQueryOptions.status()` — `staleTime: 0`, `refetchInterval: 15_000`
- `openClawQueryOptions.heartbeats()` — `staleTime: 0`, `refetchInterval: 10_000`
- `openClawQueryOptions.dailySummary()` — `staleTime: 60_000` (manual refresh)

All responses validated through their Zod schema at the call site (`.parse()`).

#### 2c — Feature module (`src/features/openclaw/`)

```
src/features/openclaw/
├─ pages/
│  └─ OpenClawPage.tsx          — four-section layout (Suspense boundaries per section)
└─ components/
   ├─ OpenClawStatusCard.tsx    — green/red badge, last-checked timestamp, auto-polls
   ├─ AgentTriggerForm.tsx      — agent_name input + optional JSON payload textarea,
   │                              submit → mutation → collapsible result/error panel
   ├─ HeartbeatMonitor.tsx      — sortable table: task_name | status badge | last_seen_at |
   │                              run_count; auto-polls every 10s; empty-state handling
   └─ DailySummaryCard.tsx      — three metric tiles (open tasks, overdue, weather placeholder)
                                   + manual "Refresh" button
```

**Key implementation notes:**
- `AgentTriggerForm`: 504 → show "Gateway timeout — not retried"; 502 → show "Connection failed" with retry allowed. Use `useMutation` with `onSettled` (not `onSuccess`).
- `HeartbeatMonitor`: `status` field maps to badge colour: `running` → blue, `success` → green, `failed` → red, unknown → grey.
- `DailySummaryCard`: weather field is a placeholder string from the API — render as-is, no special treatment.
- All components: support light + dark mode (`dark:` Tailwind prefix); use `inline style` for spacing where Tailwind v4 utility classes aren't in the bundle (follow the AppSidebar pattern).

#### 2d — Route (`src/App.tsx`)

Add lazy import:
```typescript
const OpenClawPage = lazyWithRetry(() =>
  import('./features/openclaw/pages/OpenClawPage').then((m) => ({ default: m.OpenClawPage })),
)
```

Add route inside the protected `'/'` children, consistent with other module routes:
```typescript
{
  path: 'openclaw',
  element: <ModuleGuard slug="openclaw"><OpenClawPage /></ModuleGuard>,
}
```

No prefetch loader needed (page has its own polling).
