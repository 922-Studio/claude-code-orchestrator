Fix — Deploy State Missing on Polaris After Runner Move

Runners now live on polaris and pick up jobs correctly, but every deploy workflow assumes filesystem state (/home/lab/<Service> checkouts, .env files, Docker volumes, compose projects) that physically resides on antares. Drafter PR #18 already confirms the failure mode; all other service deploys will fail identically on merge.

What we observed
  • Drafter PR-preview deploy ran on polaris-4 (good — runner labels work).
  • Step Deploy PR preview failed: /tmp/drafter-pr/pr-18/.env missing, no compose project found, no containers running.
  • Drafter deploy workflow expects a checkout at /home/lab/Drafter[-dev] on the runner host — that path only exists on antares.
  • Same pattern in HomeAPI/HomeUI/HomeAuth/HomeCollector/Studio/Portfolio/etc. — every deploy.yml references /home/lab/<Service> via reusable workflows from workflows/.github/workflows/{docker-build,smoke-test,frontend-e2e,...}.yml.
  • Beyond checkouts: Docker Swarm manager, MinIO volumes, Allure server, registry, Traefik — all run on antares. Polaris currently has Docker installed but no swarm membership state for these workloads.
Root cause

The parent plan moved runner compute to polaris without moving (or aliasing) the deploy-time state. The two are coupled: the runners were not just CI executors, they were also the host where service code, configs, and Docker state were materialised. Decoupling them needs an explicit decision.

Three remediation paths (pick one)
A

Bridge: polaris runners SSH into antares to deploy

least disruptive

Keep deploy state on antares; treat polaris runners as orchestrators that SSH into antares for filesystem and Docker operations. Requires:

  • SSH key from lab@polaris → lab@antares with passwordless sudo
  • Rewrite reusable workflows in workflows/.github/workflows/ so steps that touch /home/lab/<Service> wrap commands in ssh antares "…", or use DOCKER_HOST=ssh://lab@antares for Docker
  • Confirm runners can SSH out (Tailscale or LAN routable)

Pro: No data migration. Antares remains source of truth. Reversible.
Con: Adds an SSH hop to every deploy step; reusable workflows get more complex; runners on polaris become a thin proxy.

B

Migrate: move all service state to polaris

most aligned with the rename intent

Treat polaris as the new home for both runners and the deployed services. Antares becomes Swarm manager only (or also gets demoted).

  • Replicate /home/lab/<Service> directories from antares to polaris (rsync over Tailscale)
  • Replicate .env files (secrets management — consider sops or just rsync with care)
  • Migrate Docker volumes (named volumes via docker run --volumes-from + tar, or stop service / cp / start)
  • Move Swarm services: redeploy each stack so it lands on polaris (Swarm placement constraints or full stack redeploy from polaris)
  • Decide what stays on antares (registry, Traefik, MinIO, Allure) vs moves to polaris
  • Per-service smoke test after migration

Pro: Clean end-state. Polaris really is the new home. Antares can later be retired or repurposed as a build/CI fan-out node.
Con: Largest amount of work. Service downtime per migration. Lots of secret/volume handling.

C

Revert: dual-label antares runners with polaris

fastest unblock

Add the label polaris to the existing 4 antares runners (without re-installing or moving them). Jobs that now target runs-on: polaris route back to antares where the state actually lives. The 4 new runners on polaris stay idle or get deregistered.

  • Stop each antares runner, re-config with extra label, restart
  • OR via API: gh api -X POST /orgs/922-Studio/actions/runners/<id>/labels --field labels[]=polaris
  • Decide fate of polaris runners (deregister or keep for future migration)

Pro: Immediate unblock; rename PRs can merge today.
Con: The polaris label becomes a lie (it's not actually on polaris). Defeats the original intent of moving compute off the manager. Keep only as a stop-gap.

Decision (2026-05-23)

Chosen path: Solution A — polaris as orchestrator, antares as deploy target.

Bridge mechanism: DOCKER_HOST=ssh://lab@astro-antares for runtime deploy commands. SSH key from lab@polaris → lab@antares already provisioned and tested (DOCKER_HOST=ssh://lab@astro-antares docker ps returns prod containers).

CI state on polaris: refactor reusable workflows to use $GITHUB_WORKSPACE instead of the repository_path: /home/lab/<Service> pattern. No state seeding on polaris.

Sub-decisions for the future:

  • PR previews: polaris-local (ephemeral)
  • Registry: move from antares to polaris (separate workstream)
  • Dev / prod DBs: move to polaris (separate workstream)
Workflow refactor scope

Reusable workflows in /Users/gregor/dev/922/workflows/.github/workflows/ to refactor:

  • docker-build.yml — drop repository_path; use $GITHUB_WORKSPACE as build context. pull_code: true becomes a no-op (the runner's own checkout is the source).
  • smoke-test.yml — same: workdir = $GITHUB_WORKSPACE. env_file_path input: load from GitHub Secrets at runtime, OR pull from antares via SSH (scp lab@antares:/home/lab/<Service>/.env ./.env). Decide per-service.
  • pr-demo.yml — refactor pr-demo.sh to operate fully in /tmp/<project>-pr/pr-N/ on polaris; fetch .env via SSH from antares. Preview Traefik routing TBD (or expose via Tailscale IP tailscale_ip input, which would now point to polaris).
  • frontend-e2e.yml — same workdir change; ensure Allure URL points at the right host.
  • deploy-docker.yml — if/where it triggers prod runtime, wrap with DOCKER_HOST=ssh://lab@astro-antares. Today watchtower on antares handles the actual restart, so this may not need changes.

Per-service deploy workflows (Drafter, HomeAPI, etc.) need their with: repository_path inputs removed once the reusables stop requiring them.

Order of execution:

  3. Canary: trigger Drafter dev deploy manually; iterate until green end-to-end.

  5. Deregister antares runners.
