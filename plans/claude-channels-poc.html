discord/services/claude_runner.py — Design

ResponsibilityImplementation
Accept message + agent configasync def run(channel_id, message, agent_config) → str
Load rolling historyRedis key claude_history:{channel_id} — LRANGE 0 9 (5 pairs)
Build promptsystem persona + history text + current message, single string
Invoke claudeclaude -p "<prompt>" [--mcp-config <path>] --no-cwd via asyncio subprocess
Timeout60 s hard kill; return "[timeout — no response]" to Discord
Update historyLPUSH user/assistant pair, LTRIM to 10 items
Error handlingNon-zero exit → log stderr, return generic error string (never expose stderr to Discord)
No persistent subprocess. Each message is a fresh claude -p call. History is injected as text context, not real conversation turns. Simple, no zombie processes, sufficient for a 5-message window.

discord/cogs/claude_ai.py — Design

  • Auto-discovered by existing bot.py cog loader — no changes to bot.py needed
  • on_message listener filtered to channels in CLAUDE_AGENTS
  • Ignore messages starting with ; — preserves bot command behavior
  • Ignore messages from bots (including self)
  • Show typing indicator (async with channel.typing()) while awaiting runner
  • Post response; on error post short failure notice without exposing details

Runner Execution Model
how the bot container calls claude on the host
✓ POC choice
Option A — SSH exec
Runner calls ssh lab claude -p "..." from inside the container. No new ports or services. Uses the SSH key the container already has for server access.
production upgrade
Option B — Sidecar HTTP
Minimal FastAPI service on the host, port on host.docker.internal. More robust, easier to observe. Switch after POC is proven.
Before coding: verify the Discord bot container already has an SSH key configured for lab access. Check discord/docker-compose.yaml for volume mounts or SSH_PRIVATE_KEY env vars.
