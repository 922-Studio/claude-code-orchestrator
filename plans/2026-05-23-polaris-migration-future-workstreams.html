Polaris Migration — Future Workstreams

After the workflow refactor lands and runners are stable on polaris, three follow-up moves are needed to fully realise the "polaris = heavy lifting, antares = stable hosting" architecture. Plan only — do not execute this session.

Why these are separate

Each of these workstreams involves data migration, downtime, and credential handling. Bundling them with the runner move would have ballooned the rename effort. They're listed here so they don't get lost; each gets its own plan file when started.

Workstreams
W1

Move dev + prod databases from antares to polaris

deferred

Targets: PostgreSQL (shared_postgres, port 5432 on antares) currently hosts home_api, home_auth, home_collector, discord_bot, anime_api, drafter. Redis, MinIO metadata DBs, Allure DB likewise.

  • Per-DB: enumerate consumers, stop services, pg_dump/restore or volume rsync, update consumer connection strings to point at astro-polaris over Tailscale, restart.
  • Risk: most services reference postgres/shared_postgres by Docker network alias — moving the DB to a different host means switching from network alias to Tailscale-hostname connection strings.
  • Open question: should the DB host be a Swarm service (deployed via stack) or a plain-docker container on polaris?
  • Acceptance: each consumer service confirmed reading/writing against polaris-hosted DB; failover/backup story re-validated.

W2

Move Docker registry from antares to polaris

deferred

Current registry: registry.922-studio.com (htpasswd auth) on antares. Move to polaris.

  • Rsync registry storage volume to polaris
  • Update Cloudflare tunnel route to point at polaris
  • Update Traefik routing (if registry is fronted by Traefik)
  • Antares consumers (watchtower, manual pulls) re-test
  • Acceptance: docker push and docker pull work end-to-end against the moved registry; no manifest loss.

Future plan file: plans/2026-05-XX-registry-move-polaris.html

depends on W1 · unblocks W3
W3

PR-preview routing on polaris (preview Traefik or Tailscale-IP)

deferred

PR previews currently route through antares's Traefik via the runner's Tailscale IP. With previews moving to polaris, two routing options:

  • Tailscale-IP direct: update tailscale_ip default in pr-demo.yml from 100.112.171.16 (antares) to polaris's Tailscale IP 100.94.122.119. Previews accessible at 100.94.122.119:<port>. Simple, no Traefik on polaris.
  • Preview Traefik on polaris: deploy a second Traefik instance on polaris configured for ephemeral PR routes (e.g. pr-<N>-<repo>.preview.922-studio.com). Adds Traefik + Cloudflare tunnel split. Nicer UX, more infra.

Open question: is the preview URL going to be exposed publicly (Cloudflare) or only inside Tailscale?

Future plan file: plans/2026-05-XX-pr-preview-routing-polaris.html
