Risks & mitigations
  • Cherry-pick conflict density — four branches all rewrite invoice_service.py, invoice.html.j2, and the integration test file. Mitigation: pick smallest-delta branches last (flexible-payment is +138 lines, do it last); resolve per-file, not per-hunk.
  • Migration collisions — t001 is modified by polish; u001/u002 may assume schema state from t002. Mitigation: run alembic upgrade head + downgrade base after each cherry-pick, not only at the end.
  • HomeUI list/detail divergence — invoice-v2 and invoice-payment-actions both ship full implementations of InvoiceListPage/InvoiceDetailPage. Mitigation: treat payment-actions as the canonical version, port any v2-only UX (auto-grow iframe, branded header) onto it.
  • PR #91 conflicts with v1 PR review — if reviewers compare against #91 they'll see the same commits twice. Mitigation: close #91 immediately when v1 HomeUI PR opens (Step 05).
  • Local testing canon — per memory feedback_local_testing_vs_dev, if Gregor has been testing locally against any of the four source branches, the worktree state matters. Mitigation: confirm with Gregor before deleting any worktree containing uncommitted work.
