Worktree Bootstrap & Dev-Login Hardening

Permanently fix three recurring symptoms when running HomeUI from a fresh git worktree: blank page / "only overview", Management card visible to non-admins, and broken dev auto-login.

Root-cause findings
  • No .env.local in fresh worktrees. .env.local is gitignored. New worktrees lack it → VITE_DEV_AUTH is undefined → getDevConfig().bypass is false → no auto-login → user sees the login screen or, if a stale token exists, the dev-fallback user with roles: [] and orgPermissions: [].
  • "Only overview" symptom. HomePage renders visibleModules = [...dynamicModules, MANAGEMENT_CARD]. When /api/modules returns empty (because the user has no roles or no org), dynamicModules = [] and the page shows only the Management coming-soon card.
  • Management visible to non-admins. MANAGEMENT_CARD is unconditionally pushed into visibleModules. The same is true of the sidebar entry. Only the nested /management/organisations route is guarded via OrgManagerGuard. The parent /management route has no guard. Card & sidebar entry must check isOrgManager.
  • Silent fallback hides the real failure. applyDevFallback() sets a fake 'dev-bypass' token and isAuthenticated=true with roles:[] when the auto-login API call fails. The UI loads but every API returns 401, masking the real problem (missing env, wrong backend URL, backend down). In dev this should fail loud.
  • No documented worktree-bootstrap step. README.md Quick Start ends at npm ci && npm run dev. CLAUDE.md doesn't mention env-file copying. There is no bin/setup-worktree.sh or postinstall hook.
Strategy — separate "what's in git" from "what's per-machine"

The root cause is mixing two concerns in one ignored file. Split them:

  • Tracked, safe defaults → ship as .env.development (Vite's standard, auto-loaded by npm run dev) with non-secret dev-mode defaults: VITE_API_BASE_URL=https://lab-api-dev.922-studio.com, VITE_DEV_AUTH=admin. No credentials.
  • Per-developer secrets → stay in gitignored .env.local: only VITE_DEV_AUTH_EMAIL + VITE_DEV_AUTH_PASSWORD. Provide a tracked .env.local.example template.
  • Cross-worktree sharing → an idempotent bin/setup-worktree.sh that symlinks .env.local from the primary checkout if found, falls back to copying the example. Runs on demand and from postinstall.
  • Loud failure → in dev mode, applyDevFallback() must throw a visible banner (not a silent fake token) when auto-login fails, so the missing env file is immediately diagnosable.
  • Role gating → fix MANAGEMENT_CARD, sidebar entry, and the parent /management route to all check isOrgManager.

02

Add bin/setup-worktree.sh + npm postinstall

done

One script makes any worktree usable. Symlink-first, copy-fallback. Idempotent so it can run on every npm install.

ProjectHomeUI
DirectoryHomeUI/bin/setup-worktree.sh, package.json
Parallel with01, 03
  • Script detects whether $PWD is a worktree (git rev-parse --git-common-dir differs from --git-dir) and, if so, resolves the primary checkout's working tree (git --git-dir=$common worktree list --porcelain, take the entry without a branch-skip filter; the bare main worktree is the one whose path is the parent of .git).
  • For each of .env.local (and any other gitignored bootstrap file declared in an array at the top of the script), if missing in the current worktree and present in the primary checkout, create a relative symlink. Skip if already a symlink or regular file.
  • If no primary file exists, copy .env.local.example → .env.local and print a one-line warning telling the dev to fill in credentials.
  • Script is safe to run from a non-worktree (e.g. CI, fresh clone): in that case it only handles the .env.local.example fallback and exits 0.
  • Wire it into package.json: "postinstall": "node ./bin/setup-worktree.mjs" (use Node to avoid bash portability issues on Windows; same logic).
  • Add "setup": "node ./bin/setup-worktree.mjs" as a manual escape hatch.
  • Unit test the resolver logic with a small Vitest spec that stubs the filesystem.

Decisions to confirm before execution
  • Setup script in Node (.mjs) vs Bash. Recommendation: Node — already a project dep, runs everywhere, easier to unit-test.
  • Keep tracked .env.dev / .env.prod as-is for legacy build scripts, or migrate fully to Vite's .env.development / .env.production naming? Recommendation: migrate; one convention, less confusion.
  • Should the postinstall script run in CI? Recommendation: yes — it's a no-op when no primary checkout is found, and ensures the example file is always seeded.
  • Apply the same pattern preemptively to HomeAPI / HomeAuth worktrees, or wait until they hit the same problem? Recommendation: scope this PR to HomeUI; capture the pattern in orchestrator/guides/agent-setup so it can be ported later.
