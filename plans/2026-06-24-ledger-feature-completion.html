Current state on dev — what the analysis found
Backend ledger — solid core, integrity gaps

Confirmed dated-money model: DebtTransaction = amount + transaction_date + person_id (legacy person_name). “Times” = transaction dates, no hours/duration. Person + immutable Invoice layers exist. risk hard-delete (no audit), no POST idempotency, summary double-counts during the name→FK migration, concurrent first-debt for a new person can 500.

GSheets — backup done, reverse-sync unsafe

DB→Sheet backup done: daily 03:00 via cron_jobs, full-replace per tab, on-write debounce. Sheet→DB exists only as a generic manual upsert-by-id endpoint that skips blank-id rows, does no type coercion, and is wiped by the next backup clear(). Not viable for manual entry as-is.

MCP — exists but not personal-usable

A working FastMCP stdio server lives on antares (150 tools, 8 finance/ledger) but is server-side-only, hand-patched, not reproducible from a clean pipeline, and built against API v0.64.8 (dev is v0.71.0). Pipeline gated to prod only. No local stdio path for your Claude client.

Frontend — overhauled on dev, one gap left

re-baselined 2026-06-24 A big design-system + invoice-UI overhaul (PR #93, v0.71.0) is now merged to dev: ledger, finance overview, invoice list/detail, person-record CRUD page, and quick-add (PersonSelect/QuickAddDebtDialog/NewPersonDialog/AddDebtPage) all exist with tests. The stale local feat/finance-invoice-overhaul branch is superseded — abandon it. Remaining gap: the backend /persons/{name}/history endpoint (running balance, first/last dates) is still unused — no per-person ledger drill-down exists; the overview re-derives history client-side.

Design decisions & recommendations
  • MCP = personal-only local stdio (your choice). Run the generated FastMCP server locally against dev (then prod) with your own bearer token + X-Org-ID; register in your local Claude config. No new hosted service, no Traefik/monitoring work. Make generation reproducible so the local and prod servers are rebuildable.
  • Reverse-sync authority = app is source of truth; sheet is an append-only input surface (recommended). The per-person manual-entry sheet is a separate surface from the backup mirror and is never clear()ed. Import mints UUIDs for blank-id rows, validates+coerces types, resolves person, then writes the assigned id + a “synced ✓” marker back so rows are imported exactly once. This sidesteps the destructive-export data-loss trap entirely.
  • “100% tested” = full coverage on money-touching modules (ledger/person/invoice CRUD+routers+schemas+services, both sync directions, MCP finance tools), not 100% repo-wide. CI global gate stays 70%; the invoicing-critical paths get exhaustive unit + integration + data-loss + idempotency tests.
  • Finish the debts→ledger rename (recommended) to remove the dual-module confusion — bundled into the frontend phase; can be deferred if churn is unwelcome.
