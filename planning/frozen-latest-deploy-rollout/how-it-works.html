Frozen/Latest Rollout — How It Works

A walkthrough of what actually happens, file by file, when a deploy moves through the frozen/latest release-channel split — and which PR built which piece. Companion to the overview.html status board; that page tells you where things stand, this page tells you why.

Traced against: /Users/gregor/dev/DevOps (live repo source, 2026-07-13) Epic: DOCB-13488

0The idea in one paragraph

Every environment used to be a single moving target — merge to dev, and whatever's on dev updates immediately. "Frozen" adds a second, parallel version of each environment (dev-frozen, stage-frozen, later sandbox-frozen) that only updates when someone deliberately dispatches a deploy or promotion — a pinned lane sitting next to the moving one, same cluster, its own namespace per channel, a different Helm values overlay, and a dedicated node pool. Nothing about the application code changes for most services; it's entirely a deploy-orchestration layer bolted onto the existing pipeline.

1The four building blocks

All in the DevOps repo. Every other repo just triggers into this.

registry.yaml

One entry per service. Declares which components (api/celery/beats…) exist, and a release-channels block saying which components are allowed on which environment. This is the enforcement point — registry.py refuses to deploy anything not declared here.

services/registry.yaml

environment-meta

Maps an environment name (e.g. dev-frozen) to region, cluster, namespace, deploy-mode (pinned vs moving), and base-env (which normal environment it overlays on top of).

services/registry.yaml (environment-meta block)

3-layer Helm overlay

Every deploy layers base.yaml → values/<base-env>.yaml → values/<channel-env>.yaml, last file wins per key. The frozen file is a small delta (node pool, RELEASE_CHANNEL env var), not a full duplicate.

services/<svc>/<component>/values/*.yaml

deploy-branches.yaml

Declares which service repos get a frozen trigger at all. A generator script writes the actual on.push.branches block into each service repo's own workflow, since Actions can't read external config at trigger-eval time.

services/deploy-branches.yaml

2Flow A — merging to a service's dev-frozen branch

From a developer's merge to a running, correctly-configured pod. Nine steps.

1

Trigger is centrally declared, then mirrored into the service repo

deploy-branches.yaml says whether a service gets a frozen branch trigger at all (add-frozen: true/false). sync-build-deploy-triggers.py writes that into the service repo's own build-deploy.yaml — GitHub Actions needs the branch pattern physically in the workflow file, it can't be resolved dynamically.

scripts/sync-build-deploy-triggers.py DevOps#381
2

Merge fires the service repo's build-deploy workflow

The migration gate's first half lives here: any branch ending in -frozen skips the inline DB-migration validation job outright.

<service-repo>/.github/workflows/build-deploy.yaml — if: !endsWith(github.ref_name, '-frozen') DocBits_FTP_Import#139 (reference pattern)
3

Service repo calls into DevOps's deploy orchestrator

resolve-version job runs registry.py --repo <repo> --env dev-frozen, reads that environment's meta (deploy-mode: pinned, base-env: dev, channel: frozen), emits it as job outputs.

DevOps/.github/workflows/deploy.yml
4

Migration gate, second half — structural, not per-service

Every migration-applying job (validate-chain-eu, apply-migration-eu/us) gates on deploy_mode != 'pinned'. Frozen environments skip migrations because they're pinned, full stop — no special-casing per service needed.

deploy.yml — jobs.apply-migration-eu/us.if
5

Release-channel enforcement

enforce_release_channel() fails loudly if the service has no release-channels entry for this environment; otherwise only the declared components deploy — anything else is silently excluded (surfaced as release_channel_excluded_components in the run output).

registry.py — enforce_release_channel() DevOps#385
6

Helm deploy — 3-layer overlay applied

_helm-deploy.yml runs helm upgrade --install layering base.yaml → values/dev.yaml → values/dev-frozen.yaml. The frozen file only needs to override what actually differs.

DevOps/.github/workflows/_helm-deploy.yml DevOps#380
7

ConfigMap pre-flight check

Before the Helm upgrade, checks the expected ConfigMap already exists via kubectl get configmap — fails fast with a readable error instead of a pod crash-looping through the whole --wait timeout if envs's ConfigMap-sync hasn't caught up yet.

_helm-deploy.yml — "ConfigMap pre-flight (EU/US)" DevOps#403, #414-416
8

Node-pool placement

The frozen values delta sets nodeSelector: doks.digitalocean.com/node-pool: dev-stage-frozen — added after frozen pods were found competing with regular dev/stage traffic on the shared pool.

values/dev-frozen.yaml — nodeSelector DevOps#418
9

Image tag is pinned, not resolved from a moving ref

Because dev-frozen is deploy-mode: pinned, resolve-version requires an explicit image-tag input — no git-tag/source-env fallback. A pinned channel can never silently drift onto whatever dev currently happens to be running.

deploy.yml — resolve-version (pinned path)

3Flow B — promoting dev-frozen → stage-frozen

No rebuild. The whole point of "frozen" is that a promotion redeploys the exact same image that's already proven itself, not a new build off a moving branch.

1

Deliberately manual, not nightly

promote-dev-frozen-to-stage-frozen.yml is workflow_dispatch-only. The workflow's own header comment explains why: auto-promoting whatever happens to be sitting in dev-frozen every night defeats the purpose of a channel that's supposed to only move on deliberate hotfixes.

DevOps/.github/workflows/promote-dev-frozen-to-stage-frozen.yml DevOps#386
2

Resolve live tags off the cluster, not off git

Calls deploy-all-services.yml with read_live_tag_from: dev-frozen. resolve-live-tags.py reads each service's currently-running image tag directly out of the live dev-frozen namespace.

.github/scripts/resolve-live-tags.py
3

Same tag, redeployed to stage-frozen

Those tags feed back into deploy.yml as explicit image-tag inputs targeting stage-frozen — same overlay mechanism (base → stage → stage-frozen), same dedicated node pool, EU only (stage has no US leg).

DevOps#387 (US token fix for this exact leg)

4What RELEASE_CHANNEL=frozen actually does today

This is the part that isn't templated yet — read this before assuming it works uniformly.

The env var itself is nothing special: it's a plain line hardcoded into each service's own values/dev-frozen.yaml delta (env: [{name: RELEASE_CHANNEL, value: "frozen"}]) — registry.py doesn't inject it generically. Right now, only doc2-api actually reads it at the application level (DocBits_API/config.py, feeding org_channel_resolver.py's routing logic) — every other service just carries the label without acting on it.

Celery queue isolation for the templated services (extraction, barcode, ocr, pomatch, docnet) works completely differently: a hardcoded command: [--queues=frozen-<service>-...] override baked straight into each service's own values file — not driven by RELEASE_CHANNEL at all. registry.yaml's queue-prefix-var field exists but is explicitly reserved/unused — doc2-api's Step 4 cutover is the first place it's meant to be wired up.

6Two on-disk facts worth knowing

Auto-accounting-service's trigger config is stale on main. deploy-branches.yaml still says add-frozen: false, needs-scaffold: true for it, but registry.yaml's release-channels already declares it fully onboarded and it's live in the cluster with real image tags. The trigger-sync file was never re-run after the onboarding PR merged — cosmetic drift in the DevOps repo itself, not just in the orchestrator's docs.

DocBits_Fulltext_AI is real, not a typo. It exists as its own deploy-branches.yaml entry (add-frozen: false, needs-scaffold: true), distinct from DocBits_FullText. What it is and whether it needs onboarding is still an open question — not resolved by this pass.
